Simplyblock provides multiple security mechanisms for NVMe-over-Fabrics (NVMe-oF) connections between clients and
storage nodes. These mechanisms address access control, authentication, and data-in-flight encryption.

## Access Control: Allowed Host Lists

By default, simplyblock volumes are accessible by any host that can reach the storage node's NVMe-oF subsystem. For
environments requiring explicit access control, volumes can be configured with an **allowed host list** that restricts
connections to specific host NQNs (NVMe Qualified Names).

When an allowed host list is configured, the NVMe-oF subsystem is created with `allow_any_host=false`, and only hosts
whose NQN matches an entry in the list can connect. This is the foundational layer of NVMe-oF security.

## Authentication: DH-HMAC-CHAP (DHCHAP)

DHCHAP provides challenge-response authentication for NVMe-oF connections. It is defined in the NVMe specification
and prevents unauthorized hosts from accessing storage even if network isolation is compromised.

Simplyblock supports two authentication modes:

- **Unidirectional authentication**: The host proves its identity to the storage target. A shared secret (`dhchap_key`)
  is used in a challenge-response exchange. This prevents unauthorized hosts from connecting.

- **Bidirectional authentication**: Both host and target prove their identities to each other. In addition to the
  host key, a controller key (`dhchap_ctrlr_key`) allows the host to verify it is connecting to a legitimate storage
  target, protecting against man-in-the-middle attacks.

DHCHAP supports configurable digest algorithms (SHA-256, SHA-384, SHA-512) and Diffie-Hellman groups for key exchange.
The DH exchange adds forward secrecy to the authentication process. When the DH group is set to `null`, plain
HMAC-CHAP authentication is used without Diffie-Hellman.

Keys are automatically generated by simplyblock in the standard `DHHC-1` format and distributed to all storage nodes
hosting the volume.

## Encryption: TLS with Pre-Shared Key (PSK)

For environments where data-in-flight must be encrypted, simplyblock supports NVMe-oF TLS using pre-shared keys. TLS
encrypts all data traversing the network between the host and storage node, providing confidentiality against network
eavesdropping.

A 256-bit pre-shared key is automatically generated when TLS is enabled for a host. The `nvme connect` command
automatically includes the `--tls` flag when connecting to a TLS-protected volume.

## Combining Security Mechanisms

The three mechanisms can be combined for defense in depth:

| Level              | Mechanism            | Protection                            |
|--------------------|----------------------|---------------------------------------|
| Access control     | Allowed host lists   | Restricts which hosts can connect     |
| Authentication     | DHCHAP (bidirectional) | Verifies both host and target identity |
| Encryption         | TLS-PSK              | Encrypts data in flight               |

For production environments, combining all three provides the strongest security posture: only authorized hosts can
connect, both sides verify each other's identity, and all data is encrypted on the wire.

## Cluster-Level Configuration

DHCHAP digest algorithms and Diffie-Hellman groups are configured at the cluster level during cluster creation. This
sets the security parameters for all NVMe-oF connections within the cluster.

Per-volume security (allowed hosts, keys) is configured when creating volumes or adding hosts to volumes.
