Simplyblock provides a Kubernetes operator ([simplyblock-manager](https://github.com/simplyblock/simplyblock-manager))
that manages storage infrastructure through Custom Resource Definitions (CRDs). All CRDs are in the
`simplyblock.simplyblock.io/v1alpha1` API group.

## Common Patterns

All simplyblock CRDs follow these patterns:

- **Actions:** Many CRDs support an `action` field in the spec to trigger operations (e.g., `activate`, `failback`).
  The operator processes the action and updates the status accordingly.
- **Status subresource:** Each CRD uses a separate status subresource updated by the operator. Use `kubectl get` with
  `-o yaml` to see full status details.
- **Finalizers:** All CRDs use Kubernetes finalizers to ensure proper backend cleanup when resources are deleted.
- **Reconciliation:** Controllers requeue periodically (typically every 120 seconds) to keep status synchronized with
  the backend.

## SimplyBlockStorageCluster

Manages the lifecycle of a simplyblock storage cluster, including creation, activation, and expansion.

**API:** `simplyblockstorageclusters.simplyblock.simplyblock.io`

### Actions

| Action     | Description                                         |
|------------|-----------------------------------------------------|
| `activate` | Activates a newly created cluster                   |
| `expand`   | Initiates cluster expansion with additional nodes   |

### Spec Fields

| Field                     | Type    | Description                                                       |
|---------------------------|---------|-------------------------------------------------------------------|
| `clusterName`            | string  | Human-readable cluster name (required)                             |
| `mgmtIfc`               | string  | Management network interface name                                  |
| `stripeWdata`            | integer | Erasure coding data chunks (1, 2, or 4)                            |
| `stripeWparity`          | integer | Erasure coding parity chunks (0, 1, or 2)                          |
| `haType`                 | string  | High availability type (e.g., `dual`)                              |
| `fabric`                 | string  | Fabric type for NVMe-oF                                            |
| `blkSize`                | integer | Block size in bytes                                                |
| `pageSizeInBlocks`       | integer | Page size in blocks                                                |
| `qpairCount`             | integer | NVMe/TCP queue pair count per logical volume                       |
| `maxQueueSize`           | integer | Maximum queue size                                                 |
| `enableNodeAffinity`     | boolean | Enable node affinity for storage nodes                             |
| `strictNodeAntiAffinity` | boolean | Enforce strict node anti-affinity for HA                           |
| `isSingleNode`           | boolean | Single-node cluster mode                                           |
| `action`                 | string  | One-shot action: `activate` or `expand`                            |

### Status Fields

| Field          | Type    | Description                                                                   |
|----------------|---------|-------------------------------------------------------------------------------|
| `uuid`        | string  | Cluster UUID (auto-generated by backend)                                       |
| `status`      | string  | Cluster state: `active`, `suspended`, `degraded`, etc.                         |
| `nqn`         | string  | Cluster NVMe Qualified Name                                                   |
| `mod`         | string  | Stripe configuration (e.g., `2x1` for NDCS=2, NPCS=1)                         |
| `rebalancing` | boolean | Whether the cluster is currently rebalancing data                              |
| `configured`  | boolean | True when cluster has been successfully created in the backend                 |
| `actionStatus`| object  | Action execution state: `pending`, `running`, `success`, or `failed`           |

### Example

```yaml title="Create a storage cluster"
apiVersion: simplyblock.simplyblock.io/v1alpha1
kind: SimplyBlockStorageCluster
metadata:
  name: prod-cluster
spec:
  clusterName: prod-cluster
  blkSize: 4096
  haType: dual
  stripeWdata: 2
  stripeWparity: 1
  mgmtIfc: eth0
```

```yaml title="Activate the cluster"
apiVersion: simplyblock.simplyblock.io/v1alpha1
kind: SimplyBlockStorageCluster
metadata:
  name: prod-cluster
spec:
  clusterName: prod-cluster
  action: activate
```

### Credentials

When a cluster is created, the operator stores the cluster UUID and secret in a Kubernetes Secret named
`simplyblock-cluster-<clusterName>`. It also maintains a `simplyblock-credentials` secret for use by the CSI driver,
containing the cluster endpoint and authentication token.

## SimplyBlockStorageNode

Deploys and manages storage nodes (SPDK processes) on Kubernetes worker nodes.

**API:** `simplyblockstoragenodes.simplyblock.simplyblock.io`

### Actions

| Action     | Description                                                  |
|------------|--------------------------------------------------------------|
| `shutdown` | Shuts down a specific storage node (requires `nodeUUID`)     |
| `restart`  | Restarts a specific storage node (requires `nodeUUID`)       |
| `suspend`  | Suspends a specific storage node (requires `nodeUUID`)       |
| `resume`   | Resumes a suspended storage node (requires `nodeUUID`)       |
| `remove`   | Removes a storage node from the cluster (requires `nodeUUID`)|

### Spec Fields

| Field                       | Type      | Description                                                  |
|-----------------------------|-----------|--------------------------------------------------------------|
| `clusterName`              | string    | Name of the parent cluster (required)                         |
| `workerNodes`              | string[]  | List of Kubernetes node names to deploy SPDK on               |
| `spdkImage`                | string    | SPDK container image override                                 |
| `spdkDebug`                | boolean   | Enable SPDK debug mode                                        |
| `mgmtIfc`                  | string    | Management network interface                                  |
| `dataNIC`                  | string[]  | Network interfaces for data traffic                           |
| `maxLVol`                  | integer   | Maximum number of logical volumes per node                    |
| `maxSize`                  | string    | Maximum total provisioning size (e.g., `10Ti`)                |
| `partitions`               | integer   | Number of partitions per device                               |
| `coreIsolation`            | boolean   | Enable CPU core isolation for SPDK                            |
| `corePercentage`           | integer   | Percentage of CPU cores to isolate (1-100)                    |
| `coreMask`                 | string    | Hexadecimal CPU core mask                                     |
| `socketsToUse`             | integer   | Number of CPU sockets to use                                  |
| `nodesPerSocket`           | integer   | NUMA nodes per socket                                         |
| `pcieAllowList`            | string[]  | PCIe device addresses to use                                  |
| `pcieDenyList`             | string[]  | PCIe device addresses to exclude                              |
| `pcieModel`                | string    | PCIe device model filter                                      |
| `driveSizeRange`           | string    | Drive size range filter (e.g., `1TB-4TB`)                     |
| `useSeparateJournalDevice` | boolean   | Use a separate device for journaling                          |
| `jmPercent`                | integer   | Journal manager percentage per device (default: 3)            |
| `haJM`                     | boolean   | Enable HA journal manager                                     |
| `haJmCount`                | integer   | HA journal manager count                                      |
| `idDeviceByNQN`            | boolean   | Identify devices by NQN                                       |
| `tolerations`              | list      | Kubernetes tolerations for node placement                     |
| `action`                   | string    | One-shot node action (see Actions table)                      |
| `nodeUUID`                 | string    | Target node UUID for actions                                  |

### Status Fields

| Field          | Type    | Description                                                          |
|----------------|---------|----------------------------------------------------------------------|
| `nodes`       | array   | List of node status objects (see below)                               |
| `actionStatus`| object  | Action execution state                                                |

Each node status entry contains:

| Field              | Type    | Description                           |
|--------------------|---------|---------------------------------------|
| `uuid`            | string  | Node UUID                              |
| `health`          | boolean | Node health status                     |
| `status`          | string  | Node state (online, offline, etc.)     |
| `hostname`        | string  | Kubernetes node hostname               |
| `cpu`             | string  | CPU information                        |
| `memory`          | string  | Available memory                       |
| `volumeCount`     | integer | Number of provisioned volumes          |
| `spdkPort`        | integer | SPDK service port                      |
| `lvolPort`        | integer | LVol port                              |
| `hublvolPort`     | integer | Hub LVol port                          |
| `uptime`          | string  | Node uptime                            |

### Example

```yaml title="Deploy storage nodes"
apiVersion: simplyblock.simplyblock.io/v1alpha1
kind: SimplyBlockStorageNode
metadata:
  name: storage-nodes
spec:
  clusterName: prod-cluster
  workerNodes:
    - worker-1
    - worker-2
    - worker-3
  maxLVol: 256
  maxSize: "10Ti"
  spdkImage: simplyblock/spdk:latest
  mgmtIfc: eth0
  dataNIC:
    - eth1
    - eth2
  coreIsolation: true
  corePercentage: 80
```

The operator deploys an SPDK DaemonSet on the specified worker nodes, registers each node with the cluster backend,
and monitors node readiness.

## SimplyBlockPool

Creates and manages storage pools within a cluster.

**API:** `simplyblockpools.simplyblock.simplyblock.io`

### Spec Fields

| Field           | Type    | Required | Description                                     |
|-----------------|---------|----------|-------------------------------------------------|
| `name`         | string  | Yes      | Pool name                                        |
| `clusterName`  | string  | Yes      | Name of the parent cluster                       |
| `capacityLimit`| string  |          | Total capacity limit (e.g., `100GB`, `1TiB`)    |
| `qosIOPSLimit` | integer |          | QoS IOPS limit for the pool                      |
| `rwLimit`      | integer |          | Read/write throughput limit (MB/s)                |
| `rLimit`       | integer |          | Read throughput limit (MB/s)                      |
| `wLimit`       | integer |          | Write throughput limit (MB/s)                     |

### Status Fields

| Field    | Type   | Description                    |
|----------|--------|--------------------------------|
| `uuid`  | string | Auto-generated pool UUID        |
| `status`| string | Pool status                     |

### Example

```yaml title="Create a storage pool"
apiVersion: simplyblock.simplyblock.io/v1alpha1
kind: SimplyBlockPool
metadata:
  name: production-pool
spec:
  name: production
  clusterName: prod-cluster
  capacityLimit: "500GB"
  qosIOPSLimit: 10000
```

Deleting the CR triggers a finalizer that removes the pool from the backend.

## SimplyBlockLvol

Queries and reflects the status of logical volumes in a pool. Volumes are created via the CSI driver or management API,
not directly through this CR.

**API:** `simplyblocklvols.simplyblock.simplyblock.io`

### Spec Fields

| Field         | Type   | Required | Description                    |
|---------------|--------|----------|--------------------------------|
| `clusterName`| string | Yes      | Name of the parent cluster      |
| `poolName`   | string | Yes      | Name of the parent pool         |

### Status Fields

| Field        | Type    | Description                               |
|--------------|---------|-------------------------------------------|
| `configured`| boolean | True when pool tracking is enabled         |
| `lvols`     | array   | List of volume status objects              |

Each volume entry contains: UUID, name, node UUIDs, hostname, NQN, subsystem port, namespace ID, size, HA status,
health, encryption status, pool info, PVC name (if bound), stripe configuration, and QoS limits.

## SimplyBlockSnapshotReplication

Manages continuous snapshot-based replication from a source cluster to a target cluster for disaster recovery. Supports
automatic failover detection and manual failback.

**API:** `simplyblocksnapshotreplications.simplyblock.simplyblock.io`

### Actions

| Action     | Description                                                              |
|------------|--------------------------------------------------------------------------|
| `failback` | Triggers failback from target cluster back to source after recovery     |

### Spec Fields

| Field               | Type     | Required | Description                                                    |
|---------------------|----------|----------|----------------------------------------------------------------|
| `sourceCluster`    | string   | Yes      | Source cluster name or UUID                                     |
| `targetCluster`    | string   | Yes      | Target cluster name or UUID                                     |
| `targetPool`       | string   | Yes      | Target pool name or UUID for receiving replicated snapshots     |
| `interval`         | integer  |          | Replication interval in seconds (default: 300, minimum: 60)    |
| `timeout`          | integer  |          | Replication operation timeout in seconds                        |
| `action`           | string   |          | Set to `failback` to trigger failback                           |
| `volumeIDs`        | string[] |          | Limit replication to specific volume UUIDs                      |
| `includeVolumeIDs` | string[] |          | Volume UUIDs to include in failback (empty = all)               |
| `excludeVolumeIDs` | string[] |          | Volume UUIDs to exclude from failback                           |

### Status Fields

| Field                        | Type     | Description                                                |
|------------------------------|----------|------------------------------------------------------------|
| `configured`                | boolean  | True when replication has been configured via the backend API |
| `observedFailbackGeneration`| integer  | Tracks which generation of failback has been processed      |
| `volumes`                   | array    | Per-volume replication status (see below)                   |

Each volume entry contains:

| Field                  | Type      | Description                                                     |
|------------------------|-----------|-----------------------------------------------------------------|
| `phase`               | string    | Volume phase: `Pending`, `Running`, `Completed`, `Failed`, `Paused` |
| `lastSnapshotID`      | string    | UUID of the last replicated snapshot                             |
| `lastReplicationTime` | date-time | Timestamp of the last successful replication                     |
| `replicatedCount`     | integer   | Total number of snapshots replicated                             |
| `errors`              | array     | Timestamped error messages                                       |

### Replication Modes

The operator operates in three modes depending on cluster state:

**Normal Mode** (source cluster active):

- Periodically triggers snapshot replication at the configured interval.
- For each volume with `do_replicate=true`, creates a snapshot and transfers it to the target cluster.
- Only triggers if the previous replication task has completed.

**Failover Mode** (source cluster suspended and all nodes unreachable):

- Automatically detected by the operator -- no manual action required.
- Triggers full volume replication (`replicate_lvol`) to the target cluster as a one-time operation.
- Target volumes become primary and serve I/O.

**Failback Mode** (triggered manually via `action: failback`):

The failback process for each volume:

1. Trigger snapshot replication on the target (target-to-source) to capture changes made during failover.
2. Wait for the replication task to complete.
3. Suspend the target volume to freeze I/O.
4. Trigger a second replication to capture final delta.
5. Wait for completion.
6. Delete the target volume.
7. Notify the source cluster to resume serving the volume.

### Example

```yaml title="Set up snapshot replication"
apiVersion: simplyblock.simplyblock.io/v1alpha1
kind: SimplyBlockSnapshotReplication
metadata:
  name: dr-replication
spec:
  sourceCluster: prod-cluster
  targetCluster: dr-cluster
  targetPool: dr-pool
  interval: 300
  timeout: 600
```

```yaml title="Trigger failback after source recovery"
apiVersion: simplyblock.simplyblock.io/v1alpha1
kind: SimplyBlockSnapshotReplication
metadata:
  name: dr-replication
spec:
  sourceCluster: prod-cluster
  targetCluster: dr-cluster
  targetPool: dr-pool
  action: failback
  excludeVolumeIDs:
    - "uuid-of-volume-to-skip"
```

## SimplyBlockDevice

Queries and manages physical NVMe devices on storage nodes.

**API:** `simplyblockdevices.simplyblock.simplyblock.io`

### Actions

| Action    | Description                          |
|-----------|--------------------------------------|
| `remove`  | Removes a device from a node         |
| `restart` | Restarts a device                    |

### Spec Fields

| Field         | Type   | Required | Description                                       |
|---------------|--------|----------|---------------------------------------------------|
| `clusterName`| string | Yes      | Name of the parent cluster                         |
| `nodeUUID`   | string |          | Specific node UUID (if empty, queries all nodes)   |
| `deviceID`   | string |          | Specific device UUID for actions                   |
| `action`     | string |          | Device action: `remove` or `restart`               |

### Status Fields

| Field   | Type  | Description                                 |
|---------|-------|---------------------------------------------|
| `nodes` | array | Array of node entries, each with a `devices` list |

Each device entry contains: UUID, health, status, size, model, utilization, and performance statistics (WIOPS, RIOPS,
write/read throughput, capacity utilization).

## SimplyBlockTask

Reflects the status of background tasks running on the cluster (migrations, backups, replication, etc.).

**API:** `simplyblocktasks.simplyblock.simplyblock.io`

### Spec Fields

| Field         | Type    | Required | Description                                       |
|---------------|---------|----------|---------------------------------------------------|
| `clusterName`| string  | Yes      | Name of the parent cluster                         |
| `taskID`     | string  |          | Specific task UUID to monitor (if empty, lists all)|
| `subtasks`   | boolean |          | Include subtask information                        |

### Status Fields

| Field   | Type  | Description              |
|---------|-------|--------------------------|
| `tasks` | array | List of task entries      |

Each task entry contains: UUID, type, status, result, retry count, and timestamps.

## Operator Deployment

The simplyblock-manager operator is deployed using its own Helm chart or directly via manifests:

```bash title="Install CRDs"
make install
```

```bash title="Deploy operator"
IMG=simplyblock/simplyblock-manager:latest make deploy
```

The operator runs as a single Deployment in the cluster and watches all simplyblock CRDs. It communicates with the
simplyblock control plane API (port 5000) using Bearer token authentication stored in Kubernetes Secrets.
